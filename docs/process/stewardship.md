# Stewardship

Stewardship is recurring work that keeps the project usable: triage, docs,
issue hygiene, dependency review, release readiness, and contributor support.

Treat it as engineering work, not leftover administration.

## Triage

For incoming work or reports, decide:

- is this a bug, feature request, question, maintenance task, or design issue?
- is there enough information to reproduce or evaluate it?
- is it still relevant to supported versions or current scope?
- does it belong in a spec, design doc, note, or tracker?
- who owns the next action?

## Reproduction Standard

Bug reports should include enough context to act:

- observed behavior
- expected behavior
- steps to reproduce
- affected version or commit
- relevant logs, events, screenshots, or inputs
- whether the issue reproduces on current main

If a report is missing critical context, ask for the smallest missing piece and
mark the item as waiting.

## Stale Work

Closing stale work is not deletion. It is queue hygiene.

Define a project-specific stale policy once the project is real:

- how long to wait for reporter input
- when to close abandoned work
- when closed work can be reopened
- what labels or statuses mean

## Cadence

Pick a cadence that keeps the queue trusted:

- daily for active projects
- weekly for small maintained projects
- before each release for release-driven projects

Do a small amount regularly rather than a large cleanup rarely.
